# TODO-00: GPO Files Not Surfaced in Report File Tree

## Problem

The report's file tree dumps every file individually. On a domain with 200 GPOs, `Export-GPOAudit` generates 200+ HTML reports and 200+ XML backup directories with nested files. The GPO HTML reports -- the actually useful items -- are buried in noise. The `00-INDEX.html` (a clickable table linking to every GPO report, generated by AD's `Get-GPOReport`) already serves as the navigation hub but isn't promoted.

## Root Cause

The file tree renderer (Monarch.psm1 lines 2919-2947) groups by top-level folder only and lists every sub-path as a flat line. No mechanism exists to collapse high-cardinality generated file sets.

Commit `3b6e9d3` ("pretty file manifests") rewrote the tree renderer from hierarchical depth-based rendering to flat one-level grouping. The new CSS (`.tree-children` with left border) is better, but the flattening lost subfolder structure and exposed the cardinality problem.

## Expert Review

### Staff Engineer / Tech Lead
The fix should be a general mechanism (threshold-based folder collapse), not GPO-specific. Future high-cardinality output (e.g., per-user dormant account reports in Plan 2) should get the same treatment automatically.

### UX/UI Designer
The file tree's job is wayfinding. Individual GPO HTML filenames are meaningless -- the admin doesn't know which GPO they want until they see the index table. `00-INDEX.html` IS the interface. XML backup GUID folder names are pure noise.

Design system line 177 says "no counts, no promoted links." This was correct for v5 (handful of files). For generated file sets with their own index, a count annotation is information, not decoration -- it answers "did all my GPOs get exported?" Design system principles "data-forward" and "silence is success" support this: the count carries information, hiding 200 lines IS silence.

### SRE / Production Engineer
The recursive `Get-ChildItem` scan on a 500-GPO domain produces thousands of XML backup files. Consider counting children instead of enumerating full paths for folders that will be collapsed.

## Scope

**File:** `Monarch.psm1`, lines 2919-2947 (file tree rendering block)

### Change 1: Collapse threshold for the tree renderer

After building `$treeGroups`, add a collapse pass per subfolder group:

- Group each top-level folder's entries by their immediate subfolder
- If a subfolder group exceeds threshold (5 files): replace individual entries with:
  - The index file (`00-*`) as a separate clickable link (if present)
  - A collapsed summary line: `subfolder/* (N files)` -- non-linked, `--text-3` color
- Files not in a collapsed subfolder render normally

**Desired output for GPO section:**
```
02-GPO-Audit/
  -- 00-SUMMARY/EXECUTIVE-SUMMARY.txt
  -- 01-HTML-Reports/00-INDEX.html
  -- 01-HTML-Reports/* (47 exported GPOs)
  -- 02-XML-Backup/* (47 GPO backups)
  -- 03-CSV-Analysis/gpo-summary.csv
  -- 03-CSV-Analysis/gpo-linkage.csv
  -- 04-Permissions/gpo-permissions.csv
  -- 04-Permissions/REVIEW-overpermissioned-gpos.csv
```

### Change 2: Design system update

**File:** `docs/design-system.md`, File Tree section (lines 175-182)

Add rule for generated file sets:
- Folders with more than N sibling files collapse to a count line
- Index files (`00-*`) render as separate clickable entry above the collapse line
- Count is informational -- answers "how many items were exported?"

## What Does NOT Change

- `Export-GPOAudit` return contract -- no new properties
- `00-INDEX.html` generation -- already works correctly
- GroupPolicy domain section metrics -- separate issue
- CSS -- existing styles sufficient; collapsed line uses `--text-3`
